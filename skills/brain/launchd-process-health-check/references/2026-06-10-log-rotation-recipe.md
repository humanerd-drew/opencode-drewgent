# 2026-06-10 launchd log rotation — verified recipe

A standalone reference for rotating launchd-managed service logs. macOS newsyslog does **not** handle launchd stdout/stderr files (it only rotates syslogd output), so launchd services need their own rotation strategy.

This is the recipe that was deployed on 6/10 after a 1.7GB `gateway.error.log` accumulated from a 4-6 day outage. Verified working: archive → 9.6MB .gz; live file → 0B; service → restart picks up new FD.

---

## 1. Why this is hard

`launchd` holds the file descriptor for the service's stdout/stderr open for the entire lifetime of the process. Naive rotation strategies fail:

| Strategy | Result |
|---|---|
| `mv log log.1` + new file | launchd keeps writing to the **renamed** file. The new filename is empty forever. |
| `mv log log.1` + `kill -HUP` | launchd doesn't reopen FDs on HUP. Same problem. |
| `truncate -s 0 log` | works, but **destroys the historical log** in place. |
| `truncate -s 0` (silent) | clean, but if you want a backup, you need to copy first |

**Working pattern** (verified 6/10): **gzip to archive** + **truncate the live file** + **launchctl kickstart to reopen the FD**. The kickstart is the key — launchd kills the old process (KeepAlive respawns it), and the respawned process opens a new FD pointing to the (now empty) live file. Old archive is preserved.

## 2. The script (verbatim)

Path: `~/.hermes/scripts/drewgent_log_rotate.sh` (must be under `~/.hermes/scripts/` for Hermes `cronjob no_agent=True` — same constraint as the watchdog script).

```bash
#!/bin/bash
# drewgent_log_rotate.sh — daily rotation for launchd service logs.
# Strategy: rename .gz archive + truncate live file + launchctl kickstart.
# Cron: daily 04:00 KST, no_agent=True.

set -euo pipefail

DREW_LOGS="$HOME/.drewgent/logs"
LIB_LOGS="$HOME/Library/Logs"
MAX_SIZE_BYTES=$((100 * 1024 * 1024))   # 100MB
MAX_AGE_DAYS=7
KEEP_DAYS=30

# (label, current_log_path, restart_command)
LOGS=(
  "ai.drewgent.gateway|$DREW_LOGS/gateway.log|gateway"
  "ai.drewgent.gateway|$DREW_LOGS/gateway.error.log|gateway"
  "ai.drewgent.cron-runner|$DREW_LOGS/cron-runner.log|cron-runner"
  "ai.drewgent.cron-runner|$DREW_LOGS/cron-runner.error.log|cron-runner"
  "com.drewgent.quartz-fswatch|$LIB_LOGS/quartz-fswatch.log|quartz-fswatch"
  "com.drewgent.quartz-deploy|$LIB_LOGS/quartz-deploy.log|quartz-deploy"
  "ai.drewgent.n8n|$DREW_LOGS/n8n.log|n8n"
  "ai.drewgent.n8n|$DREW_LOGS/n8n.error.log|n8n"
)

UID_NUM=$(id -u)
TODAY=$(date +%Y-%m-%d)
rotated=()

for entry in "${LOGS[@]}"; do
  IFS='|' read -r label path service <<< "$entry"
  [ -f "$path" ] || continue

  size=$(stat -f %z "$path" 2>/dev/null || echo 0)
  age_days=$(( ( $(date +%s) - $(stat -f %m "$path") ) / 86400 ))

  rotate=0
  reason=""
  if [ "$size" -gt "$MAX_SIZE_BYTES" ]; then
    rotate=1; reason="size=${size}B > ${MAX_SIZE_BYTES}B"
  elif [ "$age_days" -gt "$MAX_AGE_DAYS" ]; then
    rotate=1; reason="age=${age_days}d > ${MAX_AGE_DAYS}d"
  fi

  if [ "$rotate" -eq 1 ]; then
    archive="${path}.${TODAY}.gz"
    gzip -c "$path" > "$archive" 2>/dev/null || true
    : > "$path"
    rotated+=("$path ($reason) → $archive")
    # Force restart so launchd reopens the FD pointing to the truncated file.
    launchctl kickstart -k "gui/${UID_NUM}/${label}" 2>/dev/null || true
  fi
done

# Cleanup archives older than $KEEP_DAYS
deleted=$(find "$DREW_LOGS" "$LIB_LOGS" -name "*.log.*.gz" -mtime "+${KEEP_DAYS}" -delete -print 2>/dev/null | wc -l | tr -d ' ')

if [ ${#rotated[@]} -eq 0 ] && [ "$deleted" -eq 0 ]; then
  exit 0   # silent when nothing to do
fi

echo "📦 **Drewgent log rotation** @ $(date '+%Y-%m-%d %H:%M:%S %Z')"
echo "Rotated: ${#rotated[@]} files"
for r in "${rotated[@]}"; do echo "  - $r"; done
echo "Deleted old archives: $deleted (kept last ${KEEP_DAYS} days)"
```

## 3. Cron registration

```
cronjob(action="create", name="Drewgent log rotation",
        no_agent=True, schedule="0 4 * * *",
        script="drewgent_log_rotate.sh")
```

Schedule: `0 4 * * *` = 04:00 KST daily. Avoids business hours, gives the system time to settle after any late-night cron jobs.

---

## 4. Tunables

| Variable | Default | When to change |
|---|---|---|
| `MAX_SIZE_BYTES` | 100MB | Lower (50MB) if disk is tight; higher (500MB) if services are chatty and rotation overhead matters. |
| `MAX_AGE_DAYS` | 7 | Lower (3) if you want to keep archives short. Higher (14) if you want a longer lookback. |
| `KEEP_DAYS` | 30 | Lower (7) if disk is tight. Higher (90) for forensic retention. |

The `(label, path, service)` tuple list is the *operational inventory* — update it when you add/remove a launchd service. Keeping it inline (not generated by discovery) makes the rotation policy reviewable at a glance.

## 5. Verification (after first run)

```bash
# 1. Confirm live file is truncated
ls -lh ~/.drewgent/logs/gateway.error.log
# expect: size ~ 0B (or small, post-rotation)

# 2. Confirm archive exists with sensible size
ls -lh ~/.drewgent/logs/gateway.error.log.*.gz | tail -3
# expect: a .gz with name like gateway.error.log.2026-06-10.gz

# 3. Confirm service is still alive (kickstart shouldn't have killed it permanently)
ps aux | grep -E "drewgent_cli.main.*gateway" | grep -v grep
# expect: PID present

# 4. Confirm new log entries appear in the live file
sleep 5
tail ~/.drewgent/logs/gateway.error.log
# expect: new log lines
```

## 6. Common pitfalls (the 6/10 fix actually hit)

1. **Forgetting the `kickstart`** — without it, launchd keeps the old FD open. The "new" file remains empty. Service logs are silently lost until next restart. **Always kickstart after truncate.**

2. **Using `truncate -s 0` instead of `: >` or `> path`** — these are equivalent in effect, but `truncate` doesn't always preserve all file metadata on macOS. `: > "$path"` is the most portable.

3. **Symlinked log paths** — some setups symlink `gateway.log` to a different file. The `gzip` archive captures the dereferenced contents correctly, but the live path after `kickstart` will be the symlink target, not the symlink itself. Verify the symlink target on the post-rotation file.

4. **Multiple services writing to the same log** — the `LOGS=()` array assumes one service per log file. If two services share (rare but possible with `StandardOutPath` collisions), rotation will break one of them. Check the plist inventory before adding entries to the array.

5. **Race condition with very high-frequency logs** — between `gzip -c` and `: >` and `kickstart`, a few lines may be written to the old FD that aren't in the archive. Loss is bounded to a few ms of log; not worth fixing. The archive is a snapshot, not a transactional snapshot.

6. **Discord webhook fail on `kickstart` (label wrong)** — if the rotation log shows `kickstart` errors with "No such process", the Label in the array doesn't match the plist's actual Label. Use `PlistBuddy -c "Print :Label" ~/Library/LaunchAgents/<file>.plist` to confirm.

7. **First run on a huge file (1.7GB case)** — `gzip -c` of a 1.7GB file takes a few seconds and uses ~1GB RAM. For larger files, consider `gzip -c | split -b 200M` for chunked archives. The 6/10 case compressed 1.7GB → 9.6MB (factor 180x) because the file was 99% repetitive warning spam.

8. **archive filename collision** — if you run the script twice in one day (manual + cron), the second run will overwrite the first archive. For a more robust pattern, use a timestamp suffix: `${path}.$(date +%Y-%m-%dT%H-%M).gz`. But the daily 04:00 cron rarely collides with a manual run.

---

## 7. When NOT to use this

- **Services that need crash-time forensics** — if the file might contain the *only* trace of a crash that hasn't fully propagated to other logs, don't rotate. Either let it grow, or move it to a separate forensics path.
- **Logs you actively tail in another terminal** — the `kickstart` will briefly interrupt the log stream (a few hundred ms). Long-running manual tailing sessions will see this as a "jump."
- **Service is currently in a tight loop of restart attempts** — if `kickstart -k` returns the service to a still-broken state, the rotation can trigger a storm. Check the service's health first.

In all these cases, prefer manual rotation: `gzip -c log > archive.gz; : > log; (no kickstart)`. The live file is truncated, the archive is saved, and the service keeps its old FD. You'll lose a few lines until the next natural restart, but the rotation is safe.
