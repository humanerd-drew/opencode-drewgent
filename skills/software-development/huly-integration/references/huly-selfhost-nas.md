# Huly Self-Host on Synology NAS — Recipie

Reference notes from the 2026-06-14/15 sessions where Drewgent self-hosted `hcengineering/huly-selfhost` on a Synology DS920+ (20 GB RAM, 16 TB free) accessed via `drew@192.168.1.53:8528` with key `~/.ssh/id_ed25519_dr2w247`. Captures the **hcengineering-standard procedure** and the gotchas that almost derailed the install.

## The standard procedure (in order)

1. **Clone the official repo** to NAS — do not improvise a `compose.yml`:
   ```bash
   cd /volume1/docker
   sudo git clone https://github.com/hcengineering/huly-selfhost.git huly
   ```
   The official compose includes cockroach's certs volume mount and the `setup.sh` + `nginx.sh` helpers. **Improvised `cr_certs_init` / `cr_huly_user` init containers are unnecessary** — hcengineering already handles this via setup.sh.

2. **Copy `example-huly.conf` to `huly_v7.conf`** (NOT `.env` — the .env is a symlink to huly_v7.conf, set up by setup.sh).

3. **Generate secrets**:
   ```bash
   sudo bash setup.sh --secret
   ```
   This creates `.huly.secret`, `.cr.secret`, `.rp.secret` (65-char hex each).

4. **Run interactive or quick setup**:
   ```bash
   sudo bash setup.sh          # interactive, prompts for HOST_ADDRESS / HTTP_PORT / SSL
   # or
   sudo bash setup.sh --quick  # uses defaults: localhost:8087
   ```
   The interactive mode prompts for **5 volume paths** — just press Enter on all 5 to use Docker named volumes (recommended). The setup also generates `nginx.conf` and runs `nginx -s reload`.

5. **Wait 60 seconds for all 14 services to initialize**:
   ```bash
   sleep 60
   sudo docker compose ps
   ```

6. **Access** at `http://<HOST_ADDRESS>` (the value you set in step 4) and sign up.

## Why `setup.sh --secret && setup.sh --quick` is a footgun

Running both chained on one line was the failure pattern. Two reasons:

- **`envsubst` is not installed on Synology DSM by default.** `setup.sh` calls `envsubst` to substitute variables into compose.yml / nginx templates. When envsubst is missing, the substitution silently fails and `${HULY_VERSION}`, `${SECRET}`, `${CR_USER_PASSWORD}` etc. all become blank strings in the running containers. Symptoms:
  ```
  WARN[0000] The "DOCKER_NAME" variable is not set. Defaulting to a blank string.
  WARN[0000] The "HULY_VERSION" variable is not set. Defaulting to a blank string.
  WARN[0000] The "SECRET" variable is not set. Defaulting to a blank string.
  WARN[0000] The "CR_DB_URL" variable is not set. Defaulting to a blank string.
  ```
  → `docker compose up -d` may "succeed" with project name = "" (validate error: `name Does not match pattern '^[a-z0-9][a-z0-9_-]*$'`), and the running containers have empty env vars.

- **The `--quick` second invocation re-sources `huly_v7.conf`**, which at that point has not been populated with secrets yet, so source errors appear: `huly_v7.conf: line 18: Self: command not found` (TITLE=`Huly Self Host` parsed as command), `line 43: syntax error near unexpected token \`newline'` (`SECRET=<setup.sh --secret>` placeholder).

### Workaround: envsubst 없이 .env 직접 채우기

`docker compose` reads `.env` directly via its own substitution. `envsubst` is only needed by `setup.sh` to template out `nginx.conf` and a few other files. For the **compose variables**, you can skip envsubst entirely by writing `huly_v7.conf` (= `.env` symlink) directly with `printf`:

```bash
cd /volume1/docker/huly
rm -f huly_v7.conf .env
printf 'HULY_VERSION=v0.7.423\nDOCKER_NAME=huly\nHOST_ADDRESS=192.168.1.53:8087\nSECURE=\nHTTP_PORT=8087\nHTTP_BIND=\nTITLE=Huly Self Host\nDEFAULT_LANGUAGE=en\nLAST_NAME_FIRST=true\nREDPANDA_ADMIN_USER=superadmin\nREDPANDA_ADMIN_PWD=<value from .rp.secret>\nCR_DATABASE=huly\nCR_USERNAME=huly\nCR_USER_PASSWORD=*** from .cr.secret>\nSECRET=*** from .huly.secret>\nDESKTOP_CHANNEL=0.7.423\nCR_DB_URL=postgresql://huly:***@cockroach:26257/huly?sslmode=disable\n' > huly_v7.conf
ln -s huly_v7.conf .env
```

**Note: indentation matters**. `cat > file <<'EOF'` heredocs with leading spaces can break — the `<<-` form only strips leading *tabs*, not spaces, and Synology's bash treats them literally. Always emit at column 0. This was a real correction from the user in this session: "네가 코드 라인을 줄때 앞에 공백 없이 주면 안되냐?"

## Required variables that setup.sh does NOT auto-inject into .env

`CR_DB_URL` is referenced by 6 places in compose.yml (`DB_URL=${CR_DB_URL}` for several services, `ACCOUNTS_DB_URL=${CR_DB_URL}` for account, `HULY_DB_CONNECTION=${CR_DB_URL}` for transactor). It is **not** auto-generated by setup.sh — you must add it manually to `huly_v7.conf` if you skip envsubst:

```
CR_DB_URL=postgresql://<CR_USERNAME>:***@cockroach:26257/<CR_DATABASE>?sslmode=disable
```

## Default port and IPs

- Web UI: `http://<HOST_ADDRESS>` — set in `HOST_ADDRESS=<ip>:<port>` (e.g. `192.168.1.53:8087`).
- Cockroach SQL: `26257` (internal, container-to-container).
- Front: `8080` (internal) → mapped to `HTTP_PORT` (e.g. 8087) via nginx.
- Each Huly service has its own internal port (account=3000, transactor=3333, collaborator=3078, etc.) — these don't need to be exposed.

## Common misconfigurations

- **Cockroach v24.2 `--insecure` + password auth**: insecure mode **disables password setting entirely** (`ERROR: setting or updating a password is not supported in insecure mode`), but the client still *prompts* for one. This deadlocks root SQL login. Fix: don't improvise `--insecure`; use the standard `start-single-node --accept-sql-without-tls` from the official compose (it's the same insecure semantics but without the password trap).
- **Cockroach certs not auto-generated**: if you find yourself writing `cr_certs_init` containers, stop. The official setup.sh handles cert generation. Check `setup.sh` source first before improvising.
- **Empty `.huly.secret` after `--quick`**: the `--quick` flag says "if .huly.secret already exists, do not overwrite." Run `--secret` once before `--quick`, or you'll get the same blank-env symptom.

## Filesystem

- `/volume1/docker/huly` — the standard install path. `/volume1` is the persistent user data partition on Synology DSM.
- `/volume1/docker/huly/.huly.secret`, `.cr.secret`, `.rp.secret` — generated secrets, 65-char hex. Do not commit to git. The repo's `.gitignore` excludes them already.
- `/volume1/docker/huly/huly_v7.conf` — the live config. `.env` is a symlink to this.

## Verification after install

```bash
# 1. All 14 containers up
sudo docker compose ps

# 2. No env-var warnings
sudo docker compose logs 2>&1 | grep -i "variable is not set" | head
# (should be empty)

# 3. Front responds
curl -s -o /dev/null -w "HTTP %{http_code}\n" http://localhost:8087

# 4. Account service healthy (no repeated DB connection errors)
sudo docker logs huly-account-1 --tail 30 2>&1 | grep -iE "error|started|listening" | tail -10
```

## What NOT to do (lessons from the 2026-06-15 struggle)

1. **Don't improvise cr_certs / cr_huly_user init containers.** The official setup.sh + example-huly.conf cover this. Adding your own init containers is a sign you haven't read `setup.sh` end-to-end yet.
2. **Don't `rm -rf /var/lib/docker/volumes/huly_*` as a "reset"**. Named volumes hold cockroach state. Use `sudo docker compose down --remove-orphans --volumes` to do it cleanly (or just let `setup.sh --reset-volumes` handle it).
3. **Don't paste the wrangler SQLITE_BUSY trace into a Huly context.** Workerd/wrangler has nothing to do with Huly. The Huly stack is Node + Mongo/Postgres + Redis, not Workers/D1. If you see wrangler errors in a Huly session, they are from a different process (likely m-log-v2 dev on port 8787). Diagnose `lsof -iTCP:8087` vs `lsof -iTCP:8787` first.
4. **Don't trust `setup.sh --quick` alone** — it requires `--secret` to have been run first AND a populated `huly_v7.conf`. Run them as separate steps.
5. **Don't `&&` chain `--secret` and `--quick`** — `--quick` re-sources `huly_v7.conf` which, after `--secret`, may not yet have all the values. Run them as separate `bash setup.sh --secret` then `bash setup.sh --quick` invocations.

## Why the user pushed back on multi-step expect automation

The agent's `expect`-based SSH automation on the NAS kept timing out (60-120s) on long chained commands. Root causes observed:
- `expect`'s `Password:` prompt match races with `sudo` password caching (5-min cache) — first sudo prompts, second sudo doesn't, and expect's `expect "drew@"` matcher picks up wrong stream chunks.
- Security policy flags `sudo docker compose down` / `sudo rm -rf /volume1/...` as destructive and blocks them outright in some agent runtimes.
- `sudo -S` (stdin pipe) is blocked as "brute-force attack vector."

**Workaround**: prefer single short commands (`sudo whoami`, `sudo docker compose ps`, `cat .secret`), and chain them across multiple `expect` invocations rather than inside one. For longer than ~3 commands, ask the user to run them in their own SSH session.
